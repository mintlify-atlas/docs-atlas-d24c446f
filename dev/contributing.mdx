## Welcome

Thank you for your interest in contributing to oxidized-cell! This guide will help you get started with contributing code, documentation, testing, and more.

<Note>
oxidized-cell is an early-stage project under active development. Many features are still being implemented, making this an excellent time to contribute.
</Note>

## Getting started

<Steps>
  <Step title="Set up development environment">
    Follow the [building from source](/dev/building) guide to set up your development environment.

    ```bash
    git clone https://github.com/darkace1998/oxidized-cell.git
    cd oxidized-cell
    cargo build
    ```
  </Step>
  <Step title="Familiarize with the codebase">
    Read the [project structure](/dev/project-structure) documentation to understand the architecture.

    Explore key crates:
    - `oc-core` - Core infrastructure
    - `oc-ppu` / `oc-spu` - CPU emulation
    - `oc-rsx` - Graphics
    - `oc-hle` - High-level emulation
  </Step>
  <Step title="Run tests">
    Ensure everything works:

    ```bash
    cargo test
    ```

    Run the emulator:

    ```bash
    cargo run --release
    ```
  </Step>
  <Step title="Pick a task">
    Check the [high-priority tasks](#high-priority-tasks) section or browse open issues on GitHub.
  </Step>
</Steps>

## Development workflow

<Steps>
  <Step title="Fork the repository">
    Click "Fork" on the [GitHub repository](https://github.com/darkace1998/oxidized-cell).

    Clone your fork:
    ```bash
    git clone https://github.com/YOUR_USERNAME/oxidized-cell.git
    cd oxidized-cell
    git remote add upstream https://github.com/darkace1998/oxidized-cell.git
    ```
  </Step>

  <Step title="Create a feature branch">
    ```bash
    git checkout -b feature/my-feature
    ```

    Branch naming conventions:
    - `feature/` - New features
    - `fix/` - Bug fixes
    - `refactor/` - Code refactoring
    - `docs/` - Documentation updates
    - `test/` - Test additions/improvements
  </Step>

  <Step title="Make your changes">
    Edit code, add tests, update documentation.

    **Before committing:**
    - Format code: `cargo fmt`
    - Run linter: `cargo clippy`
    - Run tests: `cargo test`
    - Test affected crates: `cargo test -p oc-<crate>`
  </Step>

  <Step title="Commit your changes">
    Write clear commit messages:

    ```bash
    git add .
    git commit -m "Add cellGcmSetFlipMode HLE implementation

    - Implement flip mode syscall handler
    - Add vertical blank synchronization
    - Update tests for display buffer swapping"
    ```

    Follow [Conventional Commits](https://www.conventionalcommits.org/):
    - `feat:` - New feature
    - `fix:` - Bug fix
    - `docs:` - Documentation
    - `test:` - Tests
    - `refactor:` - Code refactoring
    - `perf:` - Performance improvement
  </Step>

  <Step title="Push and create pull request">
    ```bash
    git push origin feature/my-feature
    ```

    Open a pull request on GitHub with:
    - Clear title describing the change
    - Detailed description of what and why
    - Screenshots/videos for UI changes
    - Reference related issues: "Fixes #123"
  </Step>
</Steps>

## Code style guidelines

### Rust

<Tabs>
  <Tab title="Formatting">
    **Use `rustfmt`:**
    ```bash
    cargo fmt
    ```

    **Key conventions:**
    - 4-space indentation
    - 100-character line limit
    - Use trailing commas
    - Import organization: std → external → internal
  </Tab>

  <Tab title="Linting">
    **Run `clippy`:**
    ```bash
    cargo clippy -- -D warnings
    ```

    **Address all warnings** before submitting PR.

    **Common issues:**
    - Unnecessary clones
    - Unused imports
    - Inefficient algorithms
    - Missing error handling
  </Tab>

  <Tab title="Naming">
    - `snake_case` for functions, variables, modules
    - `PascalCase` for types, traits, enums
    - `SCREAMING_SNAKE_CASE` for constants
    - Use descriptive names (avoid `x`, `tmp`)
  </Tab>

  <Tab title="Error handling">
    **Use `Result` and `?` operator:**
    ```rust
    fn load_game(path: &Path) -> Result<Game, LoadError> {
        let file = File::open(path)?;
        let elf = parse_elf(&file)?;
        Ok(Game::from_elf(elf))
    }
    ```

    **Define custom error types:**
    ```rust
    use thiserror::Error;

    #[derive(Error, Debug)]
    pub enum EmulatorError {
        #[error("Invalid memory address: {0:#x}")]
        InvalidAddress(u32),
    }
    ```
  </Tab>
</Tabs>

### C++

<Tabs>
  <Tab title="Formatting">
    **Use `clang-format`:**
    ```bash
    clang-format -i cpp/src/*.cpp
    ```

    **Settings:**
    - LLVM style
    - 4-space indentation
    - 120-character line limit
  </Tab>

  <Tab title="Standards">
    - **C++20** standard
    - Use modern features: `std::span`, concepts, ranges
    - Prefer `auto` for type inference
    - Use `constexpr` where possible
  </Tab>

  <Tab title="LLVM API">
    **Follow LLVM conventions:**
    ```cpp
    llvm::Function* create_ppu_function(llvm::Module& M) {
        auto& Context = M.getContext();
        auto* FuncType = llvm::FunctionType::get(
            llvm::Type::getVoidTy(Context), {}, false);
        return llvm::Function::Create(
            FuncType, llvm::Function::ExternalLinkage, "ppu_func", M);
    }
    ```
  </Tab>
</Tabs>

## Testing guidelines

### Writing tests

<Tabs>
  <Tab title="Unit tests">
    **Test individual functions:**
    ```rust
    #[cfg(test)]
    mod tests {
        use super::*;

        #[test]
        fn test_ppu_li_instruction() {
            let mut ppu = Ppu::new();
            ppu.execute(0x38600042).unwrap(); // li r3, 0x42
            assert_eq!(ppu.gpr[3], 0x42);
        }
    }
    ```
  </Tab>

  <Tab title="Integration tests">
    **Test component interaction:**
    ```rust
    // crates/oc-integration/tests/game_loading.rs
    #[test]
    fn test_load_and_run_elf() {
        let mut emu = Emulator::new();
        emu.load_elf("test.elf").unwrap();
        emu.run_until_halt();
        assert_eq!(emu.exit_code(), 0);
    }
    ```
  </Tab>

  <Tab title="Property testing">
    **Test with random inputs:**
    ```rust
    use proptest::prelude::*;

    proptest! {
        #[test]
        fn test_memory_access(addr in 0u32..0x1000_0000) {
            let mut mem = Memory::new();
            mem.write32(addr, 0xDEADBEEF).unwrap();
            prop_assert_eq!(mem.read32(addr).unwrap(), 0xDEADBEEF);
        }
    }
    ```
  </Tab>

  <Tab title="Benchmarks">
    **Measure performance:**
    ```rust
    use criterion::{black_box, criterion_group, criterion_main, Criterion};

    fn benchmark_ppu_execution(c: &mut Criterion) {
        c.bench_function("ppu execute 1000 instructions", |b| {
            let mut ppu = Ppu::new();
            b.iter(|| {
                for _ in 0..1000 {
                    ppu.execute(black_box(0x60000000)).unwrap(); // nop
                }
            });
        });
    }

    criterion_group!(benches, benchmark_ppu_execution);
    criterion_main!(benches);
    ```

    Run benchmarks:
    ```bash
    cargo bench
    ```
  </Tab>
</Tabs>

### Running tests

```bash
# All tests
cargo test

# Specific crate
cargo test -p oc-memory

# Specific test
cargo test test_ppu_li_instruction

# With output
cargo test -- --nocapture

# Parallel execution control
cargo test -p oc-hle -- --test-threads=1
```

## Pull request checklist

Before submitting, ensure:

<Tabs>
  <Tab title="Code quality">
    - [ ] Code follows Rust/C++ style guidelines
    - [ ] `cargo fmt` and `cargo clippy` pass
    - [ ] No compiler warnings
    - [ ] Public APIs are documented
    - [ ] Complex logic has comments
  </Tab>

  <Tab title="Testing">
    - [ ] All existing tests pass: `cargo test`
    - [ ] New features have tests
    - [ ] Bug fixes have regression tests
    - [ ] Edge cases are covered
  </Tab>

  <Tab title="Documentation">
    - [ ] README updated if needed
    - [ ] Relevant docs updated
    - [ ] API changes documented
    - [ ] Examples added for new features
  </Tab>

  <Tab title="Git history">

    - [ ] Commit messages are clear
    - [ ] No merge commits (rebase instead)

</Tabs>

## Syncing with upstream

Keep your fork updated:

```bash
# Fetch upstream changes
git fetch upstream

# Update main branch
git checkout main
git merge upstream/main
git push origin main

# Rebase feature branch
git checkout feature/my-feature
git rebase main
```

### Reporting bugs

**Include:**
- Operating system and version
- Rust/LLVM versions: `rustc --version`, `llvm-config --version`
- Steps to reproduce
- Expected vs actual behavior
- Error messages and logs
- Minimal reproduction case if possible

**Example:**
```
Title: PPU crashes on vmaddfp instruction

Environment:
- OS: Ubuntu 22.04
- Rust: 1.80.0
- LLVM: 14.0.0

Steps:
1. Load test.elf
2. Execute until PC = 0x01000420
3. Crash occurs on vmaddfp instruction

Expected: Instruction executes correctly
Actual: Panic with "invalid SIMD operation"

Logs:
[ERROR] oc_ppu: Invalid vmaddfp operands
```

## Project-specific conventions

### Memory addresses

Use hexadecimal with `0x` prefix:
```rust
const MAIN_MEMORY_BASE: u32 = 0x0001_0000;
const STACK_TOP: u32 = 0x3000_0000;
```

### Logging

Use `tracing` crate:
```rust
use tracing::{info, warn, error, debug, trace};

info!("Loading game: {}", path.display());
warn!(addr = %addr, "Unaligned memory access");
error!(err = %e, "Failed to initialize Vulkan");
debug!(gpr = ?ppu.gpr, "PPU register dump");
trace!(instr = %instr, "Executing instruction");
```

### Error messages

Be specific and actionable:
```rust
// Bad
return Err("Invalid address".into());

// Good
return Err(format!(
    "Invalid memory address {:#x}: outside valid range {:#x}-{:#x}",
    addr, MIN_ADDR, MAX_ADDR
));
```
