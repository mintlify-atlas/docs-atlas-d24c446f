# Cell Broadband Engine architecture

The PlayStation 3 is powered by the Cell Broadband Engine (Cell BE), a unique heterogeneous processor designed by Sony, Toshiba, and IBM. It combines a general-purpose PowerPC core (PPU) with 8 specialized SIMD processors (SPUs).

## Overview

<Note>
  The PS3 has 8 SPUs physically, but only 6 are usable by games. One is reserved for the OS, and one is disabled for manufacturing yield.
</Note>

## PPU (PowerPC Processing Unit)

The PPU is the main control processor, responsible for:
- Running the operating system (LV2 kernel)
- Managing game logic and AI
- Coordinating SPU workloads
- Handling I/O and system calls

### PPU architecture

**Instruction set**: PowerPC 64-bit (compatible with PowerPC 970)

**Registers**:
- **32 GPRs** (General Purpose Registers): 64-bit integer registers
- **32 FPRs** (Floating Point Registers): 64-bit double-precision floats
- **32 VRs** (Vector Registers): 128-bit VMX/AltiVec SIMD registers
- **Special registers**: LR (Link Register), CTR (Count Register), CR (Condition Register), XER (Fixed-Point Exception Register)

**Memory model**:
- 64-bit addressing (though PS3 uses 32-bit addresses)
- Big-endian byte order
- Hardware support for memory reservations (lwarx/stwcx for atomics)

## SPU (Synergistic Processing Unit)

The SPUs are specialized SIMD processors optimized for data-parallel computation:
- Graphics processing (vertex/pixel shaders)
- Physics simulation
- Audio processing
- Video decoding

<Info>
  Each SPU has 256 KB of local storage (LS) that serves as both instruction and data memory. The SPU cannot directly access main memory - it must use DMA transfers via the MFC.
</Info>

### SPU architecture

**Instruction set**: SPU ISA (completely different from PowerPC)

**Registers**:
- **128 GPRs**: All 128-bit wide, no distinction between integer/float/vector
- **Preferred slot**: Word 0 of each register is the "preferred slot" for scalar operations

**Memory**:
- **256 KB Local Storage**: Fast on-chip SRAM for code and data
- **No caches**: All memory accesses hit LS directly
- **DMA only**: MFC (Memory Flow Controller) handles transfers to/from main memory

**Channels**:
SPUs communicate via 128 channels:
- **MFC channels**: DMA commands and status
- **Mailbox channels**: Message passing to/from PPU
- **Signal channels**: Event notification
- **Event channels**: Interrupt and synchronization

### MFC (Memory Flow Controller)

The MFC is the SPU's DMA engine:

```rust
// From oc-spu/src/mfc.rs
pub struct Mfc {
    /// DMA command queue (16 entries)
    command_queue: VecDeque<DmaCommand>,
    /// Active DMA transfers
    active_transfers: Vec<DmaTransfer>,
    /// MFC tag status
    tag_status: [TagStatus; 32],
}

pub struct DmaCommand {
    pub opcode: DmaOpcode,      // get, put, etc.
    pub tag: u8,                // Tag for tracking (0-31)
    pub local_addr: u32,        // Address in local storage
    pub effective_addr: u64,    // Address in main memory
    pub size: u32,              // Transfer size
}
```

**DMA operations**:
- `get`: Transfer from main memory to local storage
- `put`: Transfer from local storage to main memory
- `getf`/`putf`: Fence variants (ensure ordering)
- `getb`/`putb`: Barrier variants (wait for completion)

## Element Interconnect Bus (EIB)

The EIB connects all Cell BE components:

<Note>
  The EIB is a ring bus with 4 data rings (2 clockwise, 2 counter-clockwise), each 16 bytes wide. Total theoretical bandwidth: 204.8 GB/s at 3.2 GHz.
</Note>

**Connected components**:
- PPU
- 8 SPUs
- Memory controller (256 MB XDR DRAM)
- I/O controller (RSX GPU, Blu-ray, USB, etc.)

**Bandwidth allocation**:
- Each component can send/receive simultaneously
- Arbitration for bus conflicts
- Optimized for streaming workloads

## Memory architecture

### Main memory (XDR DRAM)

- **256 MB** total
- **3.2 GHz** XDR DRAM (25.6 GB/s bandwidth)
- **Shared** between PPU, SPUs, and RSX (via I/O controller)

### SPU local storage

- **256 KB** per SPU (8 SPUs = 2 MB total)
- **Single-cycle access** (no cache)
- **DMA only** for main memory access

### RSX local memory

- **256 MB** GDDR3 VRAM
- **650 MHz** (22.4 GB/s bandwidth)
- **Dedicated** to RSX, accessible by PPU via MMIO

See [Memory model](/concepts/memory-model) for virtual address space layout.

## Cell BE programming model

### Task-based parallelism

Typical Cell BE program:

1. **PPU** splits work into tasks
2. **PPU** loads SPU programs into local storage via DMA
3. **PPU** starts SPU threads
4. **SPUs** process tasks in parallel
5. **SPUs** send completion events
6. **PPU** collects results

## Performance characteristics

### Theoretical peak performance

| Component | Peak Performance |
|-----------|------------------|
| PPU       | 6.4 GFLOPS (double) |
| SPU (single) | 25.6 GFLOPS (single) |
| SPUs (6 usable) | 153.6 GFLOPS (single) |
| **Total** | **~160 GFLOPS** |

### Real-world considerations

<Info>
  Achieving peak performance requires careful programming: efficient DMA usage, double buffering, and minimizing PPU/SPU synchronization overhead.
</Info>

**Bottlenecks**:
- Memory bandwidth (25.6 GB/s shared)
- DMA queue depth (16 entries per SPU)
- PPU/SPU communication latency
- Cache misses on PPU
