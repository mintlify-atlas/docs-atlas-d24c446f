# Execution modes

oxidized-cell supports multiple execution modes for both PPU and SPU code: pure interpretation, JIT compilation, and a hybrid approach. Each mode has different performance and accuracy characteristics.

## Execution mode overview

<CardGroup cols={3}>
  <Card title="Interpreter" icon="magnifying-glass">
    Decode and execute instructions one at a time. Always available, accurate, but slower.
  </Card>
  
  <Card title="JIT compiler" icon="bolt">
    Compile PowerPC/SPU code to native x86/ARM code. Fast execution, but requires C++ backend.
  </Card>
  
  <Card title="Hybrid" icon="shuffle">
    Use interpreter for cold code, JIT for hot code. Best balance of speed and memory.
  </Card>
</CardGroup>

## Interpreter mode

The interpreter decodes and executes each instruction individually:

```
Fetch → Decode → Execute → Advance PC → Repeat
```

### PPU interpreter

From `oc-ppu/src/interpreter.rs:501-577`:

```rust
pub fn step(&self, thread: &mut PpuThread) -> Result<(), PpuError> {
    // Check for breakpoints
    if self.should_break(thread) {
        return Err(PpuError::Breakpoint { addr: thread.pc() });
    }
    
    // Fetch instruction
    let pc = thread.pc() as u32;
    let opcode = self.memory.read_be32(pc)?;
    
    // Decode instruction
    let decoded = PpuDecoder::decode(opcode);
    
    // Execute instruction
    self.execute(thread, opcode, decoded)?;
    
    Ok(())
}
```

**Execution dispatch** based on instruction form:

```rust
fn execute(&self, thread: &mut PpuThread, opcode: u32, decoded: DecodedInstruction) 
    -> Result<(), PpuError> 
{
    match decoded.form {
        InstructionForm::D => self.execute_d_form(thread, opcode, decoded.op),
        InstructionForm::X => self.execute_x_form(thread, opcode, decoded.xo),
        InstructionForm::I => self.execute_i_form(thread, opcode),
        // ... more forms
    }
}
```

### SPU interpreter

Similar structure for SPU:

```rust
pub fn step(&self, thread: &mut SpuThread) -> Result<(), SpuError> {
    let pc = thread.pc();
    let opcode = thread.ls_read_u32(pc);
    
    let decoded = SpuDecoder::decode(opcode);
    self.execute(thread, opcode, decoded)?;
    
    thread.advance_pc();
    Ok(())
}
```

### Interpreter disadvantages

<Info>
  Interpreter is typically 10-100x slower than JIT for CPU-intensive code.
</Info>

- **Overhead**: Fetch, decode, and dispatch for every instruction
- **No optimization**: Instructions executed in isolation
- **Branch prediction**: Host CPU can't predict guest branches
- **Memory access**: Indirect through bounds checking

## JIT compilation

The JIT compiler translates PowerPC/SPU code to native code:

```
PowerPC/SPU code → LLVM IR → Optimized IR → Native x86/ARM code
```

### PPU JIT architecture

From `cpp/src/ppu_jit.cpp:53-195`:

```cpp
struct BasicBlock {
    uint32_t start_address;
    uint32_t end_address;
    std::vector<uint32_t> instructions;
    void* compiled_code;
    size_t code_size;

#endif

    size_t total_size;
    size_t max_size;  // 64MB default
};
```

### JIT compilation pipeline

1. **Block identification**: Find basic block boundaries
   ```cpp
   // Block ends at:
   // - Unconditional branch (b, ba, bl, bla)
   // - Conditional branch (bc, bca, bcl, bcla)
   // - Indirect branch (bclr, bcctr)
   // - System call (sc)
   // - Max 64 instructions
   ```

2. **LLVM IR generation**: Convert PowerPC to IR
   ```cpp
   llvm::Function* generate_ir(const BasicBlock* block) {
       auto* func = create_function(ctx, "block_" + hex(block->start_address));
       auto* builder = new IRBuilder(ctx);
       
       for (uint32_t instr : block->instructions) {
           emit_instruction(builder, instr);
       }
       
       return func;
   }
   ```

3. **Optimization**: Run LLVM passes
   ```cpp
   // O2 optimization passes:
   // - Constant folding
   // - Dead code elimination
   // - Loop optimization
   // - Register allocation
   ```

4. **Native code generation**: Emit machine code
   ```cpp
   void* compile_to_native(llvm::Function* func) {
       auto jit = llvm::orc::LLJIT::Create();
       jit->addIRModule(ThreadSafeModule(module, ctx));
       auto sym = jit->lookup("block_...");
       return (void*)sym->getAddress();
   }
   ```

### JIT execution

From `oc-ppu/src/interpreter.rs:289-400`:

```rust
fn try_jit_execute(&self, thread: &mut PpuThread) -> Result<bool, PpuError> {
    let pc = thread.pc() as u32;
    let mut jit = self.jit_compiler.lock();
    let jit = match jit.as_mut() {
        Some(j) => j,
        None => return Ok(false),  // JIT not available
    };
    
    // Check for compiled code
    if let Some(_compiled) = jit.get_compiled(pc) {
        // Create execution context from thread state
        let mut context = self.thread_to_context(thread);
        
        // Execute the JIT-compiled code
        match jit.execute(&mut context, pc) {
            Ok(instructions_executed) => {
                self.context_to_thread(&context, thread);
                self.jit_stats.lock().jit_executions += 1;
                return Ok(true);
            }
            Err(exit_reason) => {
                // Handle exit (syscall, breakpoint, error)
                return Ok(false);
            }
        }
    }
    
    Ok(false)  // Fall back to interpreter
}
```

#### Branch prediction

From `cpp/src/ppu_jit.cpp:998-1095`:

```cpp
struct BranchPrediction {
    uint32_t branch_address;
    uint32_t target_address;
    BranchHint hint;  // Likely, Unlikely, Static
    uint32_t taken_count;
    uint32_t not_taken_count;
    
    void update(bool taken) {
        if (taken) {
            taken_count++;
        } else {
            not_taken_count++;
        }
        
        // Update hint based on observed behavior
        if (taken_count > not_taken_count * 2) {
            hint = BranchHint::Likely;
        } else if (not_taken_count > taken_count * 2) {
            hint = BranchHint::Unlikely;
        }
    }
};
```

### JIT disadvantages

<Note>
  JIT compilation adds startup overhead and memory usage. Hybrid mode mitigates this.
</Note>

- **Compilation overhead**: Time to compile blocks before execution
- **Memory usage**: Compiled code cache (64 MB default)
- **LLVM dependency**: Requires LLVM libraries (large)
- **Complexity**: More complex implementation and debugging

## Hybrid mode

Hybrid mode combines interpreter and JIT:

```
┌─────────────────────────────────────────┐
│  Execution starts (Interpreter)         │
└────────────────┬────────────────────────┘
                 │
                 ▼
        ┌────────────────┐
        │ Execute block  │
        └────────┬───────┘
                 │
                 ▼
        ┌────────────────┐     Yes    ┌────────────────┐
        │ Hot block?     │───────────>│ Compile block  │
        │ (100+ execs)   │            │ with JIT       │
        └────────┬───────┘            └────────┬───────┘
                 │ No                          │
                 │                             │
                 ▼                             ▼
        ┌────────────────┐            ┌────────────────┐
        │ Interpret      │            │ Execute JIT    │
        │ instruction    │            │ compiled code  │
        └────────────────┘            └────────────────┘
```

### Hotness detection

From `oc-ppu/src/interpreter.rs:99-102` and `364-400`:

```rust
pub struct PpuInterpreter {
    /// Hot block threshold for hybrid mode (execution count before compilation)
    hot_threshold: u32,  // Default: 100
    /// Block execution counts for hybrid mode
    block_exec_counts: RwLock<HashMap<u32, u32>>,
}

fn try_jit_execute(&self, thread: &mut PpuThread) -> Result<bool, PpuError> {
    // ... JIT cache check ...
    
    // In hybrid mode, check if block is hot
    if mode == JitMode::Hybrid {
        let should_compile = {
            let mut counts = self.block_exec_counts.write();
            let count = counts.entry(pc).or_insert(0);
            *count += 1;
            *count >= self.hot_threshold
        };
        
        if should_compile {
            // Register for lazy compilation
            if let Ok(code) = self.read_block_code(pc) {
                jit.register_lazy(pc, &code, self.hot_threshold);
            }
        }
    }
    
    Ok(false)  // Fall back to interpreter this time
}
```

### Lazy compilation

Compilation happens in background:

```
Main thread (Interpreter)         Background thread (JIT)
       │                                  │
       │ Execute block @ 0x1000           │
       │ (count: 99)                      │
       │                                  │
       │ Execute block @ 0x1000           │
       │ (count: 100 - HOT!)              │
       │                                  │
       │ Register for compilation         │
       │─────────────────────────────────>│
       │                                  │ Compile in background
       │ Continue interpreting            │ ...
       │                                  │
       │                                  │ Compilation complete
       │<─────────────────────────────────│
       │                                  │
       │ Execute block @ 0x1000           │
       │ (JIT code now available!)        │
       │                                  │
```

### Hybrid advantages

<Info>
  Hybrid mode provides 90% of JIT performance with 10% of the memory overhead.
</Info>

- **Fast startup**: Interpreter runs immediately
- **Low memory**: Only hot code compiled
- **Adaptive**: Adjusts to workload automatically
- **Best of both**: Speed where it matters, simplicity elsewhere

## Best practices

### When to use interpreter

- **Development**: Easier debugging and testing
- **Accuracy**: When exact instruction-by-instruction behavior is needed
- **Compatibility**: Systems without LLVM or C++ compiler
- **Low memory**: Embedded systems or resource-constrained environments

### When to use JIT

- **Performance**: Production use where speed matters
- **Long sessions**: Compilation overhead amortized over time
- **CPU-intensive games**: Lots of computation, less I/O

### When to use hybrid

- **General use**: Best default for most scenarios
- **Mixed workloads**: Some hot loops, some cold initialization
- **Memory constrained**: Can't afford full 64 MB code cache
- **Quick startup**: Need to start fast but eventually run fast
